# PRD — FuturesLab (가칭)

**선물 자동매매 모의투자 툴 · 제품 요구사항 문서**
버전 0.1 (컨셉 검증) · 2026-07-15

---

## 0. 요약

코딩 못 하는 준전문 개인 트레이더가, 선물 자동매매 전략을 **노코드로 정의**하고 **바이낸스 테스트넷에 실시간 실행**해서, **청산·펀딩비 포함 실전형 성과**를 확인하는 웹 서비스.

- 대상: 한국 (1차) → 일본 (v2)
- 포지셔닝: 교육/검증 툴 (실계좌·전략추천 아님)
- 성공 정의(MVP): 사전등록 100명 + "테스트넷 연동까지 할 의향" 확인

---

## 1. 문제 정의

| # | 문제 | 근거 |
|---|---|---|
| P1 | 비개발자는 선물 봇 전략을 실전 투입 전 검증할 수단이 없다 | 현재 유일 대안이 `python-binance` 직접 코딩 |
| P2 | 백테스트만으로는 청산·펀딩비·레버리지의 실전 작동을 못 본다 | 선물 손익의 핵심이 이 셋 |
| P3 | 실계좌 봇 서비스는 자금 리스크 + 규제 노출이 크다 | 3Commas·Pionex류 |

---

## 2. 타겟 & 페르소나

**핵심 초기 타겟:** 자동매매에 관심 있지만 파이썬을 못 하는 준전문 개인 트레이더.

- 차트·지표 용어(RSI, 이평선, 변동성 돌파)는 안다
- 봇을 돌려보고 싶지만 코딩 장벽 + 돈 잃는 공포로 못 하고 있다
- 커뮤니티(비트갤, 코인판)에서 자동매매 얘기를 소비한다

**심리적 핵심:** "안전하게 먼저 돌려보고 싶다."

---

## 3. 핵심 사용자 여정 (Happy Path)

1. 가입 (Supabase Auth)
2. 바이낸스 **테스트넷 API 키 발급 안내** → 키 입력 (이 지점이 최대 이탈 구간, §7 참조)
3. 전략 템플릿 선택 (3종 중 1개)
4. 파라미터 설정 — 심볼, 레버리지, 손절/익절, 지표 값
5. **봇 실행** → 워커가 테스트넷에서 실시간 구동
6. 대시보드에서 포지션·손익·청산·펀딩비 확인
7. 결과 보고 전략 수정 → 재실행

## 4. 기능 요구사항

### 4.1 MVP (검증 단계)

| ID | 기능 | 우선순위 |
|---|---|---|
| F1 | 회원가입/로그인 (Supabase Auth) | Must |
| F2 | 테스트넷 API 키 등록 (암호화 저장) + 발급 가이드 | Must |
| F3 | 전략 템플릿 3종 (변동성 돌파 / 이평선 교차 / RSI) | Must |
| F4 | 전략 파라미터 설정 UI | Must |
| F5 | 봇 생성·시작·정지 | Must |
| F6 | 전략 실행 엔진 (워커): 캔들 종가 기준 조건 평가 → 테스트넷 주문 | Must |
| F7 | 성과 대시보드: 포지션, 실현/미실현 손익, 청산·펀딩비 반영 | Must |
| F8 | 테스트넷 한계 고지 (UX 상 명시) | Must |

### 4.2 v2 이후

- ❌ "수익 보장/기대수익률" 류 표현
- ❌ 자금 예치·운용

## 5. 시스템 아키텍처

```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│  Next.js     │────▶│  Supabase     │◀────│  Strategy Worker │
│  웹앱(Vercel) │     │  (Auth+PG)    │     │  (상시 Node 프로세스)│
└─────────────┘     └──────────────┘     └────────┬────────┘
   전략 빌더/대시보드      users/strategies/          │
                        bots/trades/positions       │
                                        ┌───────────▼───────────┐
                                        │  Binance Testnet        │
                                        │  WS(kline) + REST(order)│
                                        └─────────────────────────┘
```

### 5.1 전략 실행 엔진 (워커) — 이 제품의 심장

- 활성 봇 목록을 DB에서 로드
- 각 봇의 대상 심볼 kline WebSocket 구독
- **캔들 종가마다** 전략 조건 평가 (MVP는 tick 단위 아님 — 단순·안정 우선)
- 조건 충족 시 유저 testnet 키로 테스트넷 REST 주문 전송
- 체결·포지션·손익을 DB 기록
- 봇 정지/서버 재시작 시 상태 복구 로직 필요

> ⚠️ **이 로직은 serverless에 넣지 않는다.** 브라우저 종료와 무관하게 24시간 돌아야 하므로 상시 구동 프로세스(Railway/Fly.io/VM)에 둔다. 처음부터 워커를 웹앱과 분리한다.

### 5.2 데이터 모델 (초안)

| 테이블 | 주요 컬럼 |
|---|---|
| `users` | (Supabase auth 연동) |
| `exchange_keys` | user_id, **encrypted_api_key**, **encrypted_secret**, label |
| `strategies` | user_id, template_type, params(jsonb), name |
| `bots` | user_id, strategy_id, symbol, leverage, status(running/stopped), started_at |
| `trades` | bot_id, side, price, qty, fee, funding, executed_at |
| `positions` | bot_id, symbol, entry_price, qty, unrealized_pnl, liquidation_price |

## 6. 보안 요구사항

- **유저 테스트넷 키는 반드시 암호화 저장** (at rest). 평문 저장 금지.
- 복호화·주문 실행은 워커(서버)에서만. 키를 클라이언트로 절대 반환하지 않는다.
- `SUPABASE_SERVICE_ROLE_KEY`는 워커 전용, 클라이언트 번들 유입 금지.
- 테스트넷 키라 자금 탈취 위험은 낮지만, "키 다루는 습관"을 처음부터 올바르게 잡는다 (v2에서 메인넷 확장 시 그대로 재사용되므로).

## 7. 최대 리스크 & 검증 포인트

**최대 리스크 가설:** 타겟이 "봇 돌리고 싶다"고 말은 해도, **바이낸스 테스트넷 키를 발급받아 연동하는 허들을 넘을 의향이 있는가.**

- 이 허들이 너무 높으면 활성화율이 무너지고 제품 전체가 성립 안 함
- 완화책 후보: (a) 발급 과정 스텝별 스크린샷 가이드, (b) 공용 데모 봇으로 "연동 전 미리보기" 제공, (c) v2에서 자체 시뮬레이션 백엔드로 키 없이 체험
- **검증 계획 2주차에 최우선 확인** (사전등록자에게 "키 연동까지 할 것인가" 직접 질문)

## 8. 규제 (상세)

한국은 무기한 선물의 국내법상 정의 자체가 부재. 현행 규제는 특금법·가상자산이용자보호법 중심이며, 실질 단속선은 FIU의 "한국인 대상 한국어 영업 불법 가상자산사업자" 단속.

**설계 원칙**
1. 테스트넷 전용 — 실제 자금 절대 미연동
2. 유저 본인이 만든 전략만 실행 — 플랫폼은 추천/판매 안 함
3. 마케팅 카피에 수익 보장·기대수익률 금지
4. 서비스 성격을 "교육·검증 시뮬레이터"로 일관되게 표기

*법률 검토는 별도 필요 — 위는 제품 설계 가이드이지 법률 자문이 아님. 정식 런칭 전 전문가 검토 권장.*
